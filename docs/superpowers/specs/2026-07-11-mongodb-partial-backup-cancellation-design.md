# MongoDB 部分备份、任务取消与超时调整设计

## 目标

在保持现有 MongoDB 完全备份兼容的前提下，为 MongoDB 备份策略增加按数据库或数据库/集合排除的部分备份能力；将 Server 与备份 Agent 的备份执行超时从 1 天调整为 30 天；在备份结果中支持停止本机或远程运行中的任务，并将主动停止记录为 `cancelled`。

## 范围

本次包含：

- MongoDB 策略新建、编辑、展示与服务端校验。
- 本地 Server 和远程 Agent 的 MongoDB 部分备份执行。
- 部分备份产物的打包、压缩、加密、S3 上传、下载和保留清理衔接。
- Server 请求 Agent 及 Agent 备份子进程的 30 天超时。
- 本地和远程备份任务取消、日志状态与前端操作。

本次不包含：

- MySQL 备份范围模式变更。
- 跨 Server 或 Agent 重启恢复进程句柄或继续取消。
- 备份恢复功能或部分备份产物的自动恢复向导。
- 将 MongoDB 部分备份复用为现有 `backup_type` 的增量备份语义。

## 策略数据模型

继续使用 `BackupPolicy.extra_json` 保存 MongoDB 专属配置，不修改 `backup_type` 枚举。现有策略缺少此配置时按完全备份处理。

```json
{
  "mongo_backup": {
    "mode": "partial",
    "exclusions": [
      {"database": "logs", "collection": ""},
      {"database": "app", "collection": "audit_events"}
    ]
  }
}
```

语义如下：

- `mode=full`：完全备份，忽略并规范化清空 `exclusions`。
- `mode=partial`：部分备份，`exclusions` 至少包含一项。
- `database` 去除首尾空格后必须非空。
- `collection` 去除首尾空格后允许为空；为空表示排除整个数据库。
- 非空 `collection` 表示只排除对应数据库中的该集合。
- 不允许仅填写集合名。
- 不允许规范化后完全重复的排除项。
- 同一数据库存在整库排除项时，该库的集合级排除项属于冗余配置，服务端拒绝并提示用户删除冗余项。
- 该配置只允许用于 `db_type=mongodb`；MySQL 策略不接受 `partial` MongoDB 范围配置。

## 前端交互

MongoDB 策略弹窗增加“备份范围”单选项：

- 完全备份（默认）
- 部分备份

选择部分备份后显示动态排除列表：

- 每行包含数据库名、集合名和删除按钮。
- 数据库名必填。
- 集合名可空，并提示“留空表示排除整个库”。
- 支持新增多行。
- 保存前校验至少一行、数据库名、重复项以及整库/集合冗余冲突。
- 编辑旧策略时默认回显完全备份。

策略列表增加备份范围展示，使用户能够区分完全备份和部分备份。

备份结果列表对 `running` 记录显示“停止任务”。点击后进行二次确认，调用取消接口并刷新列表。`cancelled` 使用独立状态文案，不按失败展示。

## Server 策略校验与传输

策略创建和更新统一规范化 `extra_json.mongo_backup`。服务端承担最终校验，不能依赖前端校验。更新策略时保留 `extra_json` 中通知、S3、压缩和加密等其他配置。

远程备份提交沿用现有策略序列化，将规范化后的 `extra_json` 原样传给 Agent。本地与远程执行器读取相同字段并遵循相同语义。

## MongoDB 部分备份执行

MongoDB Database Tools 的 `--excludeCollection` 需要与单个 `--db` 一起使用，因此部分备份不能继续使用当前的一次全实例 archive 命令。部分备份流程为：

1. 使用目标实例凭据查询可备份数据库列表。
2. 过滤工具默认不备份的 `local` 数据库，并应用用户的整库排除项。
3. 为剩余数据库逐一运行 `mongodump --db=<database>`。
4. 对该数据库的每个集合排除项追加一个 `--excludeCollection=<collection>`。
5. 所有数据库写入同一个任务临时目录。
6. 所有命令成功后，将临时目录打包为一个最终备份文件。
7. 沿用现有加密、S3 上传、下载记录和保留清理流程。
8. 无论成功、失败或取消，都清理任务临时目录和未完成产物。

完全备份保持当前单 archive 行为和文件格式，不改变已有策略的执行方式。

部分备份的最终格式：

- 不压缩：`.tar`
- gzip：`.tar.gz`
- zstd：`.tar.zst`
- 启用加密后继续追加现有 `.enc` 后缀和嵌入式密钥信息。

DryRun 返回计划执行的逐库命令、跳过的数据库和最终产物路径，不启动数据库查询之外的备份子进程。若无法查询数据库列表，DryRun 返回失败，避免展示无法执行的虚假计划。

## 30 天超时

定义共享语义常量 `BACKUP_TIMEOUT_SECONDS = 2592000`，即 30 × 24 × 60 × 60 秒，并应用于：

- Server 调用 Agent 发起或等待备份相关请求的读取超时。
- Agent 直接运行 `mongodump` 的等待超时。
- Agent gzip/zstd 管道中的 `mongodump` 与压缩进程等待超时。
- 本地 Server 执行备份子进程的对应等待超时，使本地与远程行为一致。

短连接、健康检查、工具版本检查和普通轮询请求保持现有短超时，不扩大为 30 天。

## 任务取消架构

### 状态模型

`BackupLog.status` 扩展为：

- `running`
- `success`
- `failed`
- `cancelled`

用户主动停止仅写入 `cancelled`。取消完成时设置 `finished_at`，并在 `extra_json` 记录：

```json
{
  "cancelled_by_user": true,
  "cancel_reason": "cancelled by user",
  "cancel_requested_at": "<UTC ISO timestamp>"
}
```

取消不触发备份失败通知。

### 运行任务注册表

Server 和 Agent 各自维护线程安全的进程注册表：

- Server 以本地 `backup_log_id` 为键。
- Agent 以远程 `task_id` 为键。
- 注册项包含取消事件、当前活动进程集合、临时路径和最终产物路径。
- 每次启动、切换或结束子进程时，在锁保护下更新注册项。
- 任务到达任一终态后移除注册项。

每个子进程必须在独立进程组中启动。取消时先设置取消事件，再终止整个进程组；在有限等待后仍未退出则强制结束。gzip/zstd 管道中的 dump 与压缩进程均须纳入同一任务注册项。

### Server API

新增：

```text
POST /api/v1/backups/logs/<log_id>/cancel
```

行为：

- 日志不存在：404。
- 日志已是终态：200，返回当前状态，保持幂等。
- 本地运行任务：通知本地执行器取消。
- 远程运行任务：使用日志中的 Agent ID 和远程任务 ID 调用 Agent 取消接口。
- 日志为 `running` 但进程注册项不可定位：409，返回明确提示，不擅自将日志写成 `cancelled`。
- 接受取消后返回 202；执行线程负责最终清理并落库 `cancelled`。

### Agent API

新增：

```text
POST /api/v1/agent/backups/tasks/<task_id>/cancel
```

行为：

- 任务不存在：404。
- 任务已是终态：200，返回当前状态。
- 任务运行中：设置取消事件、终止进程组并返回 202。
- Agent 工作线程完成清理后将任务状态写为 `cancelled`，供 Server 后续同步。

### 竞态与清理

- 每个阶段开始前和耗时子进程结束后检查取消事件。
- 取消与自然结束并发时，最先在锁内确认的终态生效；终态不得被后续路径覆盖。
- 取消逐库备份时不再启动后续数据库命令。
- 清理未完成的 archive、tar、压缩文件、加密文件和临时目录。
- Server 或 Agent 重启后注册表丢失，取消接口返回 409；跨重启进程发现与终止不在本次范围内。

## 数据库迁移

项目使用非原生枚举约束。迁移需要更新 `backup_logs.status` 的允许值，使 `cancelled` 可持久化。迁移必须兼容现有 `running`、`success` 和 `failed` 数据，不重写历史记录。

## 错误处理

- 数据库列表查询失败：任务失败并记录可诊断错误。
- 单库 dump 失败：停止后续数据库，清理临时产物，任务标记为 `failed`。
- 打包、压缩、加密或上传失败：按现有失败路径记录，并清理不完整文件。
- 用户取消：任务标记为 `cancelled`，不记录为命令失败，不发送失败通知。
- Agent 取消请求网络失败：Server 保持日志 `running`，返回错误供用户重试；后续结果同步仍以 Agent 状态为准。

## 测试策略

### Server

- 旧 MongoDB 策略缺少范围配置时默认为完全备份。
- 完全备份规范化清空排除项。
- 部分备份拒绝空列表、空数据库名、重复项和整库/集合冗余项。
- MySQL 策略拒绝 MongoDB 部分备份配置。
- 本地部分备份生成正确的逐库命令、整库跳过和集合排除参数。
- 部分备份生成正确的 tar、gzip 和 zstd 产物路径。
- 本地取消终止活动进程、停止后续逐库命令、清理文件并写入 `cancelled`。
- 取消终态任务保持幂等；不可定位的运行任务返回 409。
- 远程取消正确转发 Agent ID 与任务 ID。
- Server 调用 Agent 的备份超时为 2592000 秒。

### Agent

- 接收并执行规范化的部分备份配置。
- 整库排除和集合排除生成正确命令。
- gzip/zstd 管道中的所有进程都受注册表管理。
- 取消接口终止进程组并最终写入 `cancelled`。
- 重复取消和终态取消保持幂等。
- 备份进程等待超时为 2592000 秒。

### 前端

- 新旧策略正确回显完全/部分备份。
- 动态添加、删除排除项并构造正确 payload。
- 空数据库、重复项和整库/集合冗余项阻止提交。
- 仅运行中日志显示停止按钮。
- 停止确认、API 调用、成功提示与列表刷新正确。

### 回归验证

- 现有 MongoDB 完全备份命令和产物格式不变。
- MySQL 完全备份不受影响。
- 压缩、加密、S3、下载、保留清理和远程结果同步测试通过。
- Backend、Agent 测试套件及 Frontend 构建通过。

## 验收标准

- 用户可在 MongoDB 策略中选择完全备份或部分备份。
- 部分备份可配置多个整库或数据库/集合排除项，并受到前后端一致校验。
- 实际产物不包含被排除的数据库或集合。
- 本地和远程备份行为一致。
- Server 与 Agent 的备份执行上限均为 30 天。
- 用户可从备份结果停止运行中的本地或远程任务。
- 主动停止最终显示为 `cancelled`，清理不完整产物且不发送失败通知。
- 完全备份和既有非相关备份功能保持兼容。
